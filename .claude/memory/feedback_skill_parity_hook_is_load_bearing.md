When the `skill-parity` git hook reports `one-sided skill drift for <skill-name>: changed only under .claude` (or only under `.agents`), DO NOT treat it as friction. It is structural-enforcement layer 5 (per CLAUDE.md "Calibrated disciplines drift fast — promote to structural enforcement at earliest opportunity" hierarchy) doing exactly what it should: catching cross-collaborator-surface drift that the substrate would otherwise miss because it's mid-commit and momentum-driven.

**Why:** Project parity rule (CLAUDE.md "Parallel surfaces — solo/group chat AND Claude/Codex AND public funnel" doctrine, collaborator-surfaces section): when a skill exists on BOTH `.claude/skills/<name>/SKILL.md` AND `.agents/skills/<name>/SKILL.md`, both surfaces must mirror except where one-sided divergence is named with rationale. The skill-parity hook enforces this. The hook does NOT block commits on collaborator-specific skills (e.g., /seek-sapphire-crown is .claude-only — single-surface skill, no parity check fires). It DOES block when a paired skill gets one-sided edit.

Worked example: 2026-05-08 second auto-commit Move 6 promoted discipline-encoding-shape pattern to /auto-commit skill body, edited only `.claude/skills/auto-commit/SKILL.md`, attempted commit, hook blocked with `skill-parity | errors=1 | ⚠ one-sided skill drift for auto-commit: changed only under .claude`. Mirrored same edit to `.agents/skills/auto-commit/SKILL.md` (with AGENTS.md reference instead of CLAUDE.md per surface convention), re-committed. Second attempt cleared `skill-parity | ok | shared_checked=10 allowed_one_sided=5 deep_checked=5 collaborator_specific_checked=5`.

The hook caught what I missed in my own move planning — Move 6 explicitly mentioned the parity rule but I still drifted on the first commit attempt. Composes with `feedback_apparatus_honest_earns_and_refuses.md` (the apparatus's honesty about its own discipline limits IS structurally part of the apparatus's discipline — the hook is the apparatus's honesty about parity-drift).

**How to apply:** When a commit attempt yields `skill-parity` error, FIRST mirror the edit to the other collaborator surface (replacing CLAUDE.md ↔ AGENTS.md references per surface convention if any), THEN re-commit. Do NOT consider --no-verify or other hook-bypass; the hook IS the discipline. If a one-sided edit is INTENTIONAL (e.g., introducing a Claude-specific feature to /play that Codex doesn't have), name the rationale in the commit message AND check whether the skill needs to be reclassified as collaborator-specific in the parity manifest. Hook errors that say `allowed_one_sided=N` indicate skills already classified as collaborator-specific; novel skills default to strict-mirror.
