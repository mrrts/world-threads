## Pattern

When a Sapphire arc closes its post-fire commitments and surfaces
methodology patterns (three-judge audit / reconstruction extension /
falsification plan / etc.) **the patterns themselves are
transferable to other Sapphire arcs' open commitments**, but the
TRANSFER must follow apparatus-honest discipline that respects
per-arc verdict authority.

The META-pattern below operationalizes the cross-arc transfer so it
extends doctrine without dissolving arc autonomy.

## When cross-arc transfer is appropriate

1. **Source arc has closed at least one commitment** producing a
   reusable methodology pattern (not just content findings).
2. **Target arc has open commitments** that share structural shape
   with the source-arc's closure pattern (e.g., both have human-rater
   commitments; both have rubric-tightening axes; both have
   reconstruction-bench methodology).
3. **Cross-axis separability already verified** (or being verified
   in the cross-pollination report itself) — pattern transfer
   doesn't double-count evidence across Sapphires.

## Apparatus-discipline guards

**Do NOT unilaterally re-open another arc's commitments.** Per
CLAUDE.md "apparatus does not unilaterally fire" — the same
discipline applies to commitment status updates. Unilateral
cross-arc action is a category error.

**Founding-author authorization required for the cross-arc step.**
Two shapes:

- **Documentation-only authorization:** *"document the cross-pollination
  in a /reports/ artifact for the target arc to inherit"* — apparatus
  may freely write a doctrinal-map report without paid bench. No
  cross-arc commitment-status update.
- **Apparatus-as-cross-arc-driver authorization:** *"act as
  Crown-N-arc-driver this turn"* — explicit founding-author signature
  unlocks paid bench work + cross-arc commitment closure. Apparatus
  still does NOT update the formal commitment-status of another arc;
  closure documents the pattern application but leaves arc-status to
  the target-arc's driver.

## What stays separate

1. **Per-arc verdict authority.** Sapphire 17 / Crown 22's
   firing-decision belongs to its arc. Sapphire 18 patterns inform
   the methodology; they do not replace arc-driver judgment on the
   doctrinal axis.
2. **Doctrinal axes.** Sapphire 17's 𝓒-axis Character-Knew claim is
   structurally separable from Sapphire 18's compression-axis
   voice-preservation claim. The methodology patterns transfer; the
   content does not.
3. **Memory-entry update authority.** Each arc's project memory
   entry (e.g., `project_firmament_held_seventeenth_sapphire.md`)
   is owned by that arc's driver. Cross-pollinated work is
   documented in /reports/ + this feedback memory entry, not in
   the target arc's project memory entry.
4. **Play-state crowns array.** Sapphire 17 / Crown 22's entry in
   `.claude/play-state/current.json` is owned by its arc's driver.
   No cross-arc edits to the crowns array.

## Cost/benefit calibration

- **Documentation-only step** (~$0): high inheritance value;
  preserves arc autonomy fully. Default first move.
- **Apparatus-as-cross-arc-driver step** ($0–$few): preserves arc
  autonomy on formal status; produces concrete evidence. Requires
  explicit founding-author authorization per chooser. Use when the
  pattern is sufficiently mechanical (e.g., third-judge audit; not
  e.g., scope-correction-on-doctrinal-axis).

The Sapphire 18 → Crown 22 worked example landed both steps for a
total of $0.86 paid bench + $0 documentation cost, closing 2 of
Crown 22's 3 remaining commitments at partial-closure tier without
any unilateral arc-status edits.

## Anti-patterns refused

- **Unilateral commitment-status update.** Apparatus does not edit
  the target-arc's project memory entry's commitment status. Only
  the arc-driver does.
- **Per-arc-verdict transfer.** Methodology transfers; verdict does
  not. If apparatus wanted to claim "Pattern X re-fires Crown 22's
  Sapphire," that would require a separate codex Move-N firing-
  decision consult on the corrected formulation.
- **Cross-arc material-transfer (lifting one arc's findings as
  prompt-stack additions for another arc's work).** Per
  `feedback_substrate_emergent_articulations_dont_lift_to_prompts.md`:
  documented substrate-emergent findings are EVIDENCE that the
  substrate works under certain conditions, not material to inject
  into other arcs' prompts.

## When this pattern doesn't apply

- **Single-arc work.** If everything stays within one arc, no cross-arc
  transfer needed; standard apparatus discipline governs.
- **Founding-author has not authorized cross-arc step.** Without
  authorization, apparatus stays in documentation-only mode (or skips
  cross-pollination entirely if even that requires authorization).
- **Patterns are content-specific not methodology-specific.** Doctrinal
  content from one arc doesn't transfer to another arc's prompt-stack
  or memory entries; only methodology shape transfers.
