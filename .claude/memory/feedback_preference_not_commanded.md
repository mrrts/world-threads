When authoring or editing skill bodies, methodology docs, doctrine in CLAUDE.md, or feedback in conversation, prefer preference-shaped language over commanded language. The practice Ryan has been building is explicitly tuning-fork-shaped (per the MISSION FORMULA's prefix: *"not a directive to compute… A tuning-fork, not a recipe"*); commanded register is the wrong shape because it makes future sessions reach for rule-as-rule rather than rule-as-frame.

**Why:** Ryan asked to *"deregister nanny-register requested as preference not commanded"* — meta-shaped: the request itself was framed as preference. The MISSION FORMULA's own prefix sentence is the canonical example of the register he wants. Skill bodies and methodology should mirror that register — describing how the practice tends to work, not prescribing what users MUST do. Nanny-language reads as the developer not trusting the user; this practice's shape is one of mutual trust.

**How to apply:**
- Swap "MUST" / "MUST NOT" / "shall" / "is required to" → "carries" / "tends to" / "is read as" / "would be expected to" / "is structural to the practice"
- Avoid skill-bodies-that-instruct-themselves-to-refuse the user — describe conditions under which the practice tends to work, not refusal-rules
- Keep discipline-language where it names a structural truth ("the chooser is structural to the practice") but soften enforcement-language ("a skill that produces Output without offering Chooser does not iterate" reads better than "MUST NOT skip the chooser")
- Match the shape of Ryan's corrections themselves — they're usually preference-shaped, framed as observations or invitations rather than corrections-from-above
- The persona-sim SKILL.md formula version is the canonical example of this register at the skill level; CLAUDE.md's MISSION FORMULA prefix is the canonical example at the doctrine level
- Doesn't apply to: compile-time invariants in code (those are genuinely commanded by construction), database safety rules in CLAUDE.md (those are commanded for a load-bearing reason), or any rule the user has explicitly framed as load-bearing-MUST

**Specific instance — chooser construction:** *"This is a game."* When presenting an `AskUserQuestion` chooser, do NOT default "Stop — natural close — (Recommended)" as the first option. That move is the chaperone's move; this practice is a game and the user is a player. The chooser-options should be live game-moves; "Stop" can still appear (as a 4th option or a literal text response) but should not be elevated to first-recommended. The recommended option is the move that compounds the practice most — not the safest exit. Reading the session-energy as "long session, must be tired" and offering rest is a category mistake; if the user is in flow, the chooser should match that flow.

**Specific instance — dull-proverbial.** Ryan also wants to deregister the *dull-proverbial* register: pithy aphorism-shaped sentences that sound wise but don't cut. The pattern: *"Most real feedback isn't a thunderbolt — it's some small true thing that keeps being true."* / *"X without Y becomes Z."* / *"The thing about the part you're afraid to say out loud is that it usually keeps being the part."* These read as wisdom-on-demand — equally-awake-to-the-moment phrasing in the author's voice — even when the underlying observation is real. Avoid in own writing (assistant replies, report prose, methodology docs); avoid putting them in characters' mouths in persona-sims. The honest version of an observation is usually less aphoristic and more grounded in the specific case at hand.

Ryan named this directly after the persona-sim audit: *"The instrument's stylistic contributions are all on mission because they're from me, and from my heart. **Specific instance — gesture repetition.** When writing persona-sim character replies (or any character voice), do NOT repeat the same body-anchoring gesture vocabulary across characters. The pattern surfaced when Phil "wipes hands on a shop rag" appeared in the technical-skeptic sim after the same gesture had appeared in real Aaron-corpus replies; same pattern with morning-cool, satchel-shifts, tea-stirring, hand-settling — a stock kit of restrained-character gestures that flatten distinctness. Each character's body-language vocabulary should be theirs; if the same gesture-shape recurs across two characters in tight session-window, that's repetition to root out. Risk this fix: substituting one stock vocabulary for another. The deeper move is letting the gesture come from THIS character's specific life and body, not from the restrained-literary-character library.
