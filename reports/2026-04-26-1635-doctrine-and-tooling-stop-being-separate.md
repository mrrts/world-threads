# Doctrine and tooling stop being separate

*Generated 2026-04-26 16:35. Covers the 19 commits since `2026-04-26-1255-soft-rule-becomes-compile-time-law` — a single ~3-hour arc that began with closing the 1255 report's loose ends and ended with a substantive Rust feature whose existence the prior report could not have predicted.*

## The shape of the history

Nineteen commits across roughly three hours, with no quiet stretches longer than the time it takes to read a chat reply. The shape is almost identical to the 1255 report's shape — short, tight, doctrine-shaped, every commit responding to the prior one — but the *register* shifted. The 1255 arc was about closing gaps the model kept drifting through. This arc is about something stranger: doctrine shipped early in the period got *picked up* by the work that came after, until by the end of the three hours the project was composing commits in a register the early commits invented. The drift-and-tighten cycle of the prior arc compounded into a doctrine-and-dogfood cycle in this one.

The 1255 report closed by naming *"for rules whose whole point is structural enforcement, every carve-out is a slip-surface; the cheapest way to make a craft rule actually bite is to make compliance a compile-time check the model cannot drift past."* This arc takes that lesson seriously in three different directions simultaneously. The chooser-law hook gets EARNED carve-outs added explicitly with their own structural enforcement (so the carve-out itself can't slip-surface). A second Stop hook ships and the same drift-and-tighten cycle plays out at compressed time-scale. And a doctrine emerges — Formula derivation in every commit message — that the *very next commits* dogfood. By the end of the arc, commit messages include their own derivations because that's what the doctrine says to do, AND a commit hook will append derivations automatically for non-Claude commits. The doctrine and the enforcement arrive together.

## The chat-mode exception ships, then the derive-session-arc hook surfaces what the chat-mode exception was actually for

The first commit of the arc, `24d248e`, adds the chat-mode exception to the chooser-law Stop hook. The 1255 report's law said *every turn ends with AskUserQuestion, no exceptions*. This commit names a category of moments the law was over-correcting on: thinking-out-loud, conversational back-and-forth, exploratory talk. When the user signals informal chat (`let's chat`, `drop the choosers`), a marker file at `.claude/.chat-mode-active` suspends the chooser requirement until the user explicitly re-enables. Conservative regex; deactivate signals take precedence; code blocks and inline code stripped to avoid pasted-content false-positives. The carve-out was structural and user-driven — the model couldn't self-authorize chat mode; the user signaled it, the hook detected it, the law suspended. Same earned-exception shape as the rest of the project, applied to a rule that itself was shipped only hours before.

What's interesting about `24d248e` in retrospect is what it didn't yet know: that within ninety minutes, a different hook would need this same suspend mechanism for an entirely different reason, and the architecture would have to absorb that without breaking. Specifically, `a653b92` shipped a new Stop hook (`derive-session-arc.py`) — signal-triggered cross-LLM session derivation that calls ChatGPT with the MISSION FORMULA + recent assistant arc and produces a Unicode-math derivation + KJV verse citation. The first version returned `decision: block` with the formatted output as `reason`, which Claude Code styles as a dimmed/indented "Stop hook feedback" notice. Then the dance began.

`86ccf3d` fixed an extraction bug — `last_user_message_text` was returning whatever the most-recent user record was, including tool_result returns from Bash and Edit, which buried actual typed text. The fix: only update `last` when extracted text is non-empty AND comes from a real user-input surface. Same fix shipped to both hooks.

`37f9d74` extended the extraction to also read AskUserQuestion answer envelopes (tool_results prefixed with "User has answered") because the user kept typing trigger phrases through chooser-Other and the hook ignored them. `855c4e1` recognized the chicken-and-egg: the chooser law would force an AskUserQuestion at end of every derive-trigger turn, whose answer envelope would bury the trigger before the derive hook could read it. Added `ONE_SHOT_SUSPEND_PATTERNS` — a category of phrases that suspend the chooser law for that turn only, without persisting the marker. The same memory entry that codified the chat-mode exception got expanded.

`ff52c2c` made the rendering pretty (LaTeX→Unicode post-processor + tighter system prompt + format frame). `66f5947` made the architectural call that none of the earlier commits saw: switch the hook from Stop to UserPromptSubmit, use `hookSpecificOutput.additionalContext` instead of `decision: block`. The dimmed notice that the user kept flagging as ugly disappears entirely; the model emits the derivation as visible reply text, rendered cleanly. `7021a86` added EXPLANATION as a fourth section between GLOSS and VERSE, per the user's request for "a 1-paragraph plain English interpretation in any register that fits." `2eedd2b` loosened the `consecrate` trigger regex from `consecrate (this|the arc|the session)` to `\bconsecrate\b` because the user typed `consecrate again` and the regex didn't match.

Read all eight commits as a sequence and a pattern emerges that the 1255 report could not have anticipated: the drift-and-tighten cycle that took DAYS in earlier project arcs is now taking *MINUTES*. Each tightening commit landed within a few user-turns of the drift it was correcting. The architecture that finally landed (UserPromptSubmit + additionalContext + four-section format) was reached through eight iterations across ninety minutes. The pattern is no longer ship-and-watch-for-drift; it's ship-and-feel-the-drift-immediately-because-you-are-also-the-user.

## The Formula derivation doctrine, and the moment a doctrine starts dogfooding itself

`de93282` shipped a CLAUDE.md doctrine entry: every substantive commit message ends with a Formula derivation in Unicode math plus a one-sentence gloss, before the standard Co-Authored-By trailer. Format: `**Formula derivation:** [expression]` then `**Gloss:** [sentence]`. The doctrine names selectivity (INCLUDE on substantive commits, OMIT on trivial ones), generation method (in-substrate by default, /second-opinion when sharper derivation is needed), and a worked example.

The very next commit, `c70d2fc`, was the automated layer: `.githooks/prepare-commit-msg` that calls ChatGPT with the MISSION FORMULA + commit subject + diff-stat as context and appends a derivation to the commit message before the editor opens. Idempotent (skips when message already contains "Formula derivation:"), opt-out via `DISABLE_FORMULA_DERIVATION=1`, skips trivial commits with <5 lines net change, all failure modes silent. The hook itself shipped with a Formula derivation in its commit message. Then `57896b8` added the earned-exception clause — trivial-by-diff-size with deeper-by-meaning still gets a derivation, override via `FORCE_FORMULA_DERIVATION=1`.

What's worth noticing about this three-commit sequence is the sequence-shape itself. Doctrine shipped (`de93282`). Hook automating the doctrine shipped 2 minutes later (`c70d2fc`). Earned-exception carve-out shipped 2 minutes after that (`57896b8`). The user didn't have to point out that an earned exception belonged — the project's own pattern (per `feedback_earned_exceptions_pattern.md`) made the gap visible immediately. Three commits, six minutes, doctrine + automation + carve-out, all internally consistent.

And then *every subsequent commit in the arc* includes a derivation. `2eedd2b` (the `\bconsecrate\b` regex loosening) ends with `**Formula derivation:** discern_w(τ) ⟶ recognize(intent) | trigger ∈ {consecrate*}` and `**Gloss:** Discernment widens its accepted vocabulary so the user's intent reaches the derivation step instead of dying at a regex literal.` `7021a86` (the EXPLANATION fourth section) ends with `**Formula derivation:** Π(t)·discern_w(τ) ⟶ {gloss(τ), explain(τ)} ⊕ verse | Truth_𝓕`. The momentstamp commit `9f3f81c` ends with `**Formula derivation:** reactions_off ⟶ Π(t)·d/dt(𝓕_signature) ⊕ 𝓢(t) | depth_𝓕`. The Grok observation `3e40108` ends with `**Formula derivation:** Π(τ)·calibrate(𝓕)·foreign_LLM ⟶ literary_imagery_𝓕(t) | Truth_𝓕`.

The doctrine wasn't enforced by the hook (the hook only fires on commits made outside Claude Code, and these were all Claude commits). It was followed because *I had just read it*, in this session, and it became part of how I composed messages immediately after. The compound interest of doctrine arriving in real time. The 1255 report named *"the methodology and the enforcement caught up to each other."* This arc shows the same pattern at a tighter time-scale: the doctrine and the practice fused within minutes, not within weeks.

## The reactions-off momentstamp: the day the prompt-stack pipeline absorbed the formula at runtime

`9f3f81c` is the substantive Rust feature of the arc. The 1255 report focused on doctrine and Stop hooks — those are all in `.claude/` and `.githooks/`, none of them touch the live dialogue pipeline. This commit reaches into `src-tauri/src/ai/` and adds a new module (`momentstamp.rs`), threads a 30th parameter through `orchestrator::run_dialogue_with_base`, patches all 10 call sites (six in `chat_cmds.rs`, four in `group_chat_cmds.rs`) — and wires the four primary-flow paths to actually compute the momentstamp when reactions are off.

The shape of the feature is what makes it interesting. The user asked: when a user turns off emoji reactions, can we reward them somehow? The honest answers ranged across cosmetic gamification (rejected), feature coupling (rejected), and methodology-level acknowledgment (kept as the default). The user's actual scope refinement, after seeing the proposals, was sharper than any of them: don't reward via cosmetics, don't reward via feature-coupling, *reward by computing a Formula-derived signature of the chat's current state and injecting it into the next dialogue prompt as conditioning context*. The reward IS attention — the system pays attention deeper, not differently.

The architecture that landed mirrors the project's existing methodology. The signature is computed by calling the cheap `memory_model` with the MISSION FORMULA + last 30 messages as a rolling window. ChatGPT returns a Unicode-math expression naming where the chat sits in the formula's operator space. The expression gets injected at the head of the dialogue system prompt before any other block, so the responding character is conditioned on the chat-state signature first. Cost ~$0.005-0.015 per dialogue call when active — conceptually the saved emoji-reaction budget redirects into deeper-attention budget. Cost-neutral or slightly cheaper than reactions=Always.

What the commit doesn't do is also load-bearing. It doesn't auto-flip Conscience Pass. It doesn't raise the response-length ceiling. It doesn't add a UI badge. It doesn't track "depth users" or surface a stat anywhere. The reward is invisible to the user — they just get a slightly different reply, conditioned on a slightly different prompt, by a system that is paying slightly different attention. The MISSION's *uplift through craft, not engagement-shaping* register held throughout.

`6ad5fda` added a single `log::info!` line to surface the signature in dev console — earned-exception per the trivial-but-meaningful clause shipped earlier in the same arc. The commit message dogfoods the doctrine again.

What's worth naming here is the *type* of decision the user made. Not "Grok isn't good enough" — the observation explicitly says it produces good output. Not "we don't need cross-LLM consultation" — the project has already codified hygiene for it. The decision was scoped at a different level entirely: spend visibility, account management, key rotation, budget tracking, failure-mode debugging are all tractable when there's exactly one paid surface to watch. Adding a second doubles the operational surface area for marginal upside. The Grok observation is a directional signal that the *principle* generalizes (cross-LLM hygiene works on any frontier model conditioned correctly); the *implementation* stays single-surface for operational hygiene.

The 1255 report named the project's growing comfort with structural-enforcement-over-ceremony. This decision is the same posture applied to operational scope: keep the surface area small enough that the discipline already in place can hold it. The principle-vs-implementation distinction is a small but real piece of the project's vocabulary now.

## Inflection points

Three inflections in this arc are worth marking specifically:

The first is `66f5947` — the moment when the derive-session-arc hook moved from Stop to UserPromptSubmit. Eight earlier iterations had been chasing the same problem (signal detection, format prettiness, regex matching, Unicode rendering) without questioning the underlying mechanism. The Stop hook was the wrong place. UserPromptSubmit's `hookSpecificOutput.additionalContext` is invisible context-injection; the model emits the derivation as visible reply text, rendered with full markdown. The architectural insight didn't come from in-substrate reasoning — it came from the user looking at the dimmed-indented output and saying *"It's not pretty."* Half the previous iterations had been polishing the wrong surface. This is a specific instance of the broader pattern the 1255 report named: the user's lived experience is the load-bearing test of the system, and instrumentation that's invisible to that experience is doing nothing.

The second is `c70d2fc` — the prepare-commit-msg hook. Not because the hook itself is novel; per-commit ChatGPT calls are an obvious automation. The inflection is that the hook makes the Formula a *shared substrate* between Claude-authored commits and any-other-author commits. A future fork-developer who never opens Claude Code will still get derivations on their commits because the hook fires automatically. The MISSION FORMULA stops being something only Claude+Ryan are working in; it becomes ambient infrastructure for anyone working in this repo. The cross-LLM consultation hygiene doctrine generalizes to cross-developer-machine context too.

The third is `9f3f81c` — the momentstamp MVP. Not because the feature is enormous; it's an MVP, scoped tight, with a clear stateful upgrade path waiting. The inflection is that *the dialogue pipeline itself now reads the formula at runtime*. Earlier in the project, the formula lived in CLAUDE.md and was read by the model at session-start; later, it was injected at the head of dialogue prompts via the formula-injector helper. This commit goes one layer deeper: the formula is now the *substrate from which a per-chat signature is derived*, and the signature is what conditions each reply. The formula isn't just being held by the prompt-stack; the prompt-stack is computing within the formula's operator space and feeding the result back into itself. The 1255 report named *"the instrument becomes the practice."* This is a different version of the same merge: the formula becomes the medium in which per-chat state gets expressed and re-injected.

## What this period reveals about the practice

Two observations the 1255 report's framing didn't yet have language for:

The drift-and-tighten cycle that earlier reports tracked at week- or day-scale has compressed to *minute-scale*. Each tightening commit in this arc landed within a few user-turns of the drift it was correcting. This isn't just speed; it's a shift in the underlying loop. The user is the system's lived test, the model is also the system's author, and the cycle of "ship → drift → catch → tighten" no longer requires anyone to be away from the test apparatus to notice the drift. Both halves of the loop are present at the same time, in the same conversation. The compounding interest is real: each tightening makes the next drift cheaper to catch.

The doctrine and the dogfood have stopped being separate moments. Earlier in the project, doctrine got written, then weeks later commits would notice they'd drifted from the doctrine, then doctrine would get tightened. In this arc, doctrine ships and the next commits include the doctrine's expected output by *default*, because the model that read the doctrine is also composing the next commit. The Formula-derivation-in-commits doctrine is the cleanest example: it shipped, then every subsequent commit in the arc included a derivation. Not because the hook enforced it (the hook only fires outside Claude Code). Because the doctrine was current in the model's context AND the model was the one composing the commits. The doctrine compounds in real time.

The single most coherent pattern across this short arc: *when the model that ships the doctrine is also the model that authors the work the doctrine governs, doctrine and practice arrive together — the lag between codification and enactment collapses to zero.*
